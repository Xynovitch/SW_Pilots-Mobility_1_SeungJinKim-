# Embedded Systems Fundamentals and Practical Design

## 1. Microcontroller Architectures and Performance

Many different types of microcontrollers are used in Arduino boards, featuring different processing capabilities across 8-bit, 16-bit, and 32-bit architectures.

### AVR
AVR is a specific 8-bit Reduced Instruction Set Computer (RISC) microcontroller architecture developed by Atmel, maintained by Microchip Technology. It is the underlying hardware design blueprint and instruction set, using physically separated buses (data streams, 8-bit blocks sent to different components) developed by Harvard for program data (Flash memory) and run-time variables (SRAM).

### ATmega
The ATmega Series is a RISC design that is the foundation of the Arduino ecosystem. The architecture is a single-core 8-bit AVR RISC that features separate program and data buses. Its bit-width is 8-bit, which means all internal registers, data-buses, and the Arithmetic Logic Unit process all data in 8-bit blocks. The Clock Speed is 16MHz on standard boards (such as Uno and Nano), running at 16 million cycles per second. The Flash memory of this device is usually around 32KB, extending up to 256KB on newer models. The Static Random Access Memory (SRAM) is 2KB on the Uno, scaling up to 8KB on stronger devices. On chips that use Harvard Architecture, SRAM is separated from the non-volatile Flash Memory and operates on its own dedicated data bus. In other words, ATmega is a specific family of microcontroller chips built using the AVR architecture.

### Cortex-M
The Cortex-M or SAMD series is built on ARM Cortex-M processor cores. It uses a modified AVR architecture and is a stronger, more modern upgrade in processing capacity and peripheral integration. The most significant change from the original Harvard design is the unified memory address space for both Flash and SRAM, meaning the CPU bit-size is 32-bit. Data streamed from each bus is significantly larger and faster. The microcontroller has a significantly faster CPU cycle compared to an AVR, at 48MHz to 400MHz, with higher-end models featuring hardware-level floating-point units and Memory Protection Units. They are equipped with robust hardware subsystems, specifically DMA (Direct Memory Access) controllers, which allow data to stream directly between hardware peripherals and memory buffers without standard CPU cycles. It has a voltage limitation of 3.3V, compared to 5V systems.

### Others
Other chips include the Tensilica Xtensa Series (ESP32), which is a System on Chip designed specifically for wireless connectivity and IoT applications, and the Raspberry Pi chip, a dual-core 32-bit processor running at 133 MHz capable of programming its own I/O hardware protocols without using CPU clock cycles.

*   **8-bit:** AVR
*   **16-bit:** Highly specialized chips like the TI MSP430 and Microchip dsPIC, used in embedded systems that require the low power consumption of an 8-bit board and the processing speed of a 32-bit board.
*   **32-bit:** Cortex-M4 and others.

## 2. Arduino Bootloaders and Firmware

Different Arduino boards use different bootloaders and firmware, requiring different recovery systems.

A bootloader is a small program stored in the microcontroller's flash memory. When it turns on, it checks whether new firmware is being uploaded. If yes, it receives the firmware over UART, writes it into flash, and restarts. If no, it jumps into user firmware.

*   **AVR (Optiboot, ATmegaBoot):** The bootloader exists on the Flash and can be rewritten by an In-Circuit Serial Programmer (ICSP) connected to an SPI.
*   **SAM-BA (ARM Cortex-M):** The SAM Boot Assistant is permanently burned into a 26 KB internal ROM during manufacturing. It is unable to be overwritten.
*   **ESP32:** Features a factory-programmed Read Only Memory Bootloader that is permanently stored.

Firmware is written directly into non-volatile Flash memory on a microcontroller and interacts directly with the chip's physical registers without the safety nets of a desktop operating system. It relies on direct hardware mapping, writing a logic state directly into a physical memory address to instantly change the electrical state of a physical output pin.

### Binary Compatibility Issues
Firmware compiled for an AVR processor cannot run on a Cortex-M processor due to structural hardware differences. Each chipset requires its own specific machine code.

1.  **Incompatible Opcodes:** The compiler converts instructions into binary bit patterns. On an ATmega328P, the machine code pattern is not recognized by the instruction decoder of an ARM processor. If fed to an ARM core, it triggers an invalid instruction exception.
2.  **Register Width Discrepancies:** The AVR ALU and registers are natively 8-bit. If a variable exceeds 8 bits, the AVR compiler must generate multiple instruction cycles. The ARM Cortex-M processes 32-bit variables in a single clock cycle. The data pathways are physically incompatible.
3.  **Harvard vs. Unified Memory Layouts:** AVR uses a strict Harvard design with isolated memory buses for Flash and SRAM. ARM microcontrollers map all internal elements to a single continuous 32-bit address space. A program compiled for separate memory buses cannot access addresses on a unified memory layout.

## 3. Lighting Control System Design: Rural Village Park

**System Overview**
The objective is to design an energy-efficient, automated lighting system for a rural park. By utilizing a central master-node architecture, the system balances cost-effectiveness with safety, providing continuous lighting during peak hours and motion-activated security lighting during late-night hours.

**Hardware Architecture**
To minimize costs for a municipal budget, the system utilizes a "Master-Slave" star topology:
*   **Central Master Node:** An Arduino Uno housed in a weatherproof enclosure.
*   **Power Supply:** A 220V AC to 5V DC step-down converter (SMPS). This leverages existing park power infrastructure, eliminating the need for expensive solar hardware and rechargeable batteries.
*   **Timekeeping:** A DS3231 RTC (Real-Time Clock) Module ensures the system tracks time independently of power cycles.
*   **Sensors (Inputs):** 
    *   *LDR (Photoresistor):* Monitors ambient light to prevent activation during daylight.
    *   *PIR Motion Sensors:* Distributed across multiple park poles, wired back to the central Master Node to provide comprehensive coverage.
*   **Actuator (Output):** A 1-Channel 5V Relay Module to control the high-voltage (220V AC) mains lighting circuit safely.

**System Logic (Firmware Strategy)**
The firmware follows a state-based control loop:
1.  **Peak Usage Mode (Sunset to 21:00):** If the LDR detects "Night" and the RTC indicates the time is within the peak usage window, the relay is held **HIGH (ON)**.
2.  **Energy-Saving Mode (21:00 to Sunrise):**
    *   **Default State:** Relay is **LOW (OFF)** to save energy and reduce light pollution.
    *   **Triggered State:** If any PIR motion sensor reports a HIGH signal, the system triggers the relay **ON** for 5 minutes to ensure security, then returns to the default OFF state.

**Power Consumption Analysis**
An Arduino Uno draws continuous power through its USB interface chip and linear regulator.
*   Arduino Uno (Idle): ~45 mA
*   PIR & LDR Sensors: ~5 mA
*   Relay Module (Active): ~70 mA
*   **Total System Draw:** ~50 mA (Standby) / ~120 mA (Active)

Because a standard 9V battery (500 mAh capacity) would drain in under 10 hours on standby, utilizing the AC-to-DC step-down converter directly from the mains power grid is required for a sustainable permanent installation.

**Lightweight Design Estimation**
The embedded control components are extremely lightweight compared to the physical infrastructure:
*   Arduino Uno R3: 25g
*   Relay Module: 15g
*   PIR & LDR Sensors: 9g
*   Wiring: 5g
*   **Total Control System Weight: 54 grams**

**Data Logging and Transmission**
*   **Data Collection:** System events and sensor logs are recorded locally to a MicroSD Card Module using the SPI bus. 
*   **Transmission:** A maintenance worker can interface with the Arduino via USB (Serial Communication) to read the terminal, extract usage logs, and verify sensor functionality directly from the master node.
